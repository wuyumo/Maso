# Coach 多模态输入 — 设计文档 (只设计, 未实现)

> 状态: 提案 (2026-07-10)。本文档盘点 Coach composer 之外的输入模态 (图片 / 链接 / 视频),
> 给出三条路线与落地顺序。当前版本只有文本 + "#" 模板半填空; 本文所列均未开工。

## 0. 现状盘点

照片导入链路**已通**:

- **QR 深链无损导入** — 分享卡二维码 → `maso://` 深链 → 完整 Plan JSON 还原, 零损耗。
- **OCR 照片导入** — Vision 本地 OCR 识别训练表照片 → `RoutineReviewSheet` 人工校对 → 入库。
- **入口**: Coach composer `[+]` 菜单 "Import from photo" + Today "All" sheet 的 "+" 菜单。

局限: 这条链路的语义是「**导入成 routine**」, 不是「**给 coach 当上下文**」——
用户拍一张别人的计划想让 AI "参考着改", 目前没有通路。

## 1. 路线① 截图 → Coach 上下文 (最快)

**思路**: OCR 出来的文本不再走"导入"分支, 改作 `coachGenerate` 的 `feedback`,
与 CURRENT ROUTINES 一起送 LLM — 用户语义 = "按这张图改我的计划"。

- 复用现成 Vision 本地 OCR, **原图不上传**, 只有识别出的文字进 prompt。
- 入口: `[+]` 菜单加一项 "Photo as context" (或在现有导入流末尾给分叉:
  「导入成计划 / 发给教练参考」)。
- 改动面: CoachScreen 一个入口 + DataStore 把 OCR 文本拼进 feedback; Worker 无需改。

**工期估**: 1–2 天。

## 2. 路线② 链接 (视频 / 文章)

**思路**: 用户把 YouTube/B站/文章链接丢给 coach → 提取动作清单 → 修订环。

管道:

1. **iOS Share Extension** 收链接 (系统分享面板 → Maso)。
2. **Cloudflare Worker 加抓取路由** — 服务端拉取链接内容:
   - YouTube / B站 **只能读字幕/标题/简介, 看不了画面** (Worker 是文本管道);
   - 字幕缺失时降级到 标题 + 章节列表。
3. LLM 从文本里**提取动作清单** (动作名/组次/顺序)。
4. `matchExercise` 把动作名映射到 938 动作库 (canonical 精确命中优先)。
5. 进现有**修订环**: 生成 DAY 卡 → 用户继续对话微调。

**依赖**: Worker 改造 (新路由 + 抓取/字幕解析), 是主要工作量。
**工期估**: 3–5 天。

## 3. 路线③ 视频画面理解 (远期)

- 帧抽样 + 视觉模型 (看画面识动作), **成本高** (推理贵 + 上传流量), 远期再议。
- **折中**: 引导用户把训练笔记/白板拍成照片走路线① OCR — 覆盖大部分真实需求。

## 4. 落地顺序

| 顺序 | 路线 | 工期 | 备注 |
|---|---|---|---|
| 1 | ① 截图→上下文 | 1–2 天 | 纯客户端, 复用现成 OCR |
| 2 | ② 链接 | 3–5 天 | 依赖 Worker 改造 (抓取路由 + 字幕) |
| 3 | ③ 视频画面 | 远期 | 成本/收益不成立, 先用①折中 |

## 5. 隐私边界

- **图片本地识别** (Vision on-device), 原图永不上传; 只有 OCR 文本进 prompt。
- **链接内容** 由 Worker 服务端转成文字后才进 prompt; 客户端不直接抓第三方页面。
- 聊天文本按现行规则属 PII: analytics 只报长度/来源面, 不报内容。
