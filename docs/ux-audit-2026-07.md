## 一、已核验 P0/P1(23 条)

### 1. [P0] 「AI Workout Planner」核心承诺全链路空转: AI 生成的计划永远不上屏, 上屏计划的重量既不看画像也不读训练历史

**发现路**: onboarding, domain

**证据**: 三段链路互相锁死: ① 引导过渡页真实等待 LLM (DataStore.swift:1196-1210), 但 generateFirstPlanViaAI 先 seedStarterRoutines 种 2 条本地模板, AI 结果只写 aiTodayPlan; TodayScreen.swift:48-53 `suggested = todayRecommendedPlan ?? aiTodayPlan`, 而 pickTodayPlan (RecommendedPrograms.swift:281-307) 只要 plans 非空必返回一条——引导刚种了 2 条, aiTodayPlan 分支对新用户永不可达。② 上屏模板重量硬编码 (RecommendedPrograms.swift:77-92: 深蹲 80kg/RDL 70kg), tunedRecommendedPlans 只改 sets/reps 不改 weight (DataStore.swift:416-424)——60kg 女新手答完 7 问 day-1 见 80kg 杠铃深蹲, 而性别/体重步副标题承诺 'set sensible starting weights' (OnboardingScreen.swift:166/245)。③ 即便 AI 路径可达也是盲猜: buildAIPayload 的 recentHistory 不含任何 weight/reps (DataStore.swift:1714-1756), LLM 直接猜 weight_kg (AIWorkoutService.swift:756-762), 且 refreshAIWorkoutIfNeeded 每天重置用户手动调过的重量 (DataStore.swift:1148-1167)。附带: RootView.swift:263/273 每天静默调一次 DeepSeek 生成这个永不展示的计划, 纯烧钱。练 5 年的人打开'今日 AI 训练'看到卧推 55kg (实际 100kg), 一眼判死这是玩具; Fitbod 的立身之本恰恰是按历史负重做 progression。

**建议**: ① 首日 (plans 里只有 autoGenerated 种子时) 让 suggested 优先 aiTodayPlan 并带 ✨AI 标, 或把 AI 结果落成 plans 第一条; ② buildAIPayload 附近 14 天每动作 best set, prompt 要求基于该记录做 0~+2.5kg 微进阶; ③ 客户端兜底: buildPlan 匹配库内动作后用 lastSet.weight 覆盖 LLM 猜值; ④ fallback 模板按体重/经验系数折算 (深蹲新手 ≈0.5×BW), 做不到就删掉引导里的承诺文案; ⑤ 维持现状则停掉每日静默 LLM 调用止损。

### 2. [P0] 动作库被静默截断到前 200 个: 默认浏览里引体向上、推举、高位下拉、绝大多数深蹲全部不存在, 自创动作永远被切掉

**发现路**: library

**证据**: ExerciseLibraryBrowser.swift:153 `Array(arr.prefix(200))` 对 filter 结果硬截断, 且截断发生在 JSON id 字母序上——705 个非 niche 动作实测: overhead_press 13 个全在 idx 422+ 被切光、pull_up 16 个全在 idx 460+ 被切光、lat_pulldown 12 个全灭、squat 47 个只剩 3 个、Shoulders 区只剩 14/98; 选 Muscle=Legs (233 个) 同样超限被切。自创动作 append 在 705 个之后 (DataStore.swift:95), 默认浏览永远看不到自己刚建的动作。UI 无任何'仅显示前 200'提示; 训练中选动作 picker 同一写法 (PlansScreen.swift:1746) 问题成对出现。用户会直接断定'这 app 连引体向上都没有'——一行代码级的即弃用 bug。

**建议**: 直接去掉 prefix(200): List 是 lazy 的, 705 行不构成性能问题。若坚持上限: 截断前先按 displayName 排序、自创+置顶永远保留、列表末尾给'已显示 200/705'行。随下一版必修。

### 3. [P0] 从 Classics 保存多日计划只会静默存下第一天, 却顶着整套计划的名字——存了 PPL 六天实际只拿到 Push 日

**发现路**: plans

**证据**: ClassicsSheet.communityDisplayPlan (PlansScreen.swift:436-440) 只取 cp.materialize(byId:).first 再改回整套项目名; 加号和详情页 Save 都走 addToSaved → savePlan(单张) (PlansScreen.swift:443-447)。materialize (CommunityPlans.swift:4896-4930) 明确返回每 session 一张 Plan; 旧 CommunityScreen.handleAdd (CommunityScreen.swift:247-260) 是 append 全部 sessions 的正确语义, 但该屏已不可达 (TodayScreen.swift:36 communityPresented 无任何代码置 true)。附带误导: 卡片上 'PPL 6-day' 显示的动作/组数只是 Push 日一天的量, 与标题的 6-day 自相矛盾。用户以为自己在跑 PPL, Pull/Legs 日被静默丢弃且毫无感知——发现时必然认定 app 丢数据。

**建议**: 恢复 handleAdd 语义: save 时 materialize 全部 sessions 一起 append (命名 '项目名 · Session 名'), 成功 toast '已添加 3 个训练日'; 详情页列出全部 session 或至少标注'含 N 个训练日', 让所见与所存一致。

### 4. [P0] 全局动作参数同步默认开且零场景内告知——改一处静默改写所有计划, 同一动作的重/轻日周期化在默认设置下根本做不出来

**发现路**: plans

**证据**: Settings.swift:188 `globalExerciseParamSyncEnabled = true` 默认开; DataStore.updatePlan diff 后调 syncExerciseParams (DataStore.swift:693-741), 把 sets/reps/weight/rest 连同逐组覆盖 setReps/setWeights 全部写进所有 plan + aiTodayPlan; 训练中在播放器临时调重同样触发传播 (PlanPlayerScreen.swift:1334-1338)。唯一披露藏在 Settings > Exercise data 一段长描述 (SettingsScreen.swift:174-179), 编辑页和详情页零提示。双重打击: ① 重日 5×5@100 / 轻日 3×12@70 (极常见编排) 任何一次编辑或训练中调整都会把另一天覆盖掉; ② 用户打开没碰过的 routine 发现组数重量全变, 只会认为 app 丢数据/有 bug。

**建议**: 默认改 OFF (同步作 opt-in, 新增动作仍从历史回填——这才是用户想要的 90% 场景); 至少: 首次实际跨计划传播时弹一次性说明 ('已同步到另外 2 个含卧推的计划——可在设置关闭') + EditStepView 参数区常驻一行'同步所有计划中的该动作'小字可点跳设置。

### 5. [P0] 组记录默认失真且永不可修正: ✓ 记的是计划值而非实际完成值, 误触无撤销 (撤销功能已是死代码), 事后只能删整场训练

**发现路**: training, usability

**证据**: 三个缺口叠加: ① advance() 写 SetRecord 直接取计划值 (TrainingSession.swift:396-405)——计划 8 次力竭只做 6, 想记 6 需铅笔→sheet→开 Different per set→调→Save ≥5 次点击且必须在 ✓ 之前想起来; 周 5 练用户最后一两组常态化力竭, 容量/PR 统计系统性虚高。② ✓ 点完主按钮原位立刻变'跳过休息' (PlanPlayerScreen.swift:1604-1615), 手抖双击=记组+跳休息; 回退键恒为 skipBackToPrevExercise 只移播放头不清记录 (PlanPlayerScreen.swift:999-1001); undoLastCompletedSet (TrainingSession.swift:559) 和 DataStore.removeLastSet (817) 全仓无调用方=撤销通道被整个删掉; 最后一组误触直接推 completed 态弹完成页, 无路可回。③ HistoryScreen.swift:308-320 只有删除整场 session, 单组记录永不可改。对以数据自诩的健身 app, 日志既不准又改不了是弃用级; Strong/Hevy 均可当场取消/事后改单组。

**建议**: ① ✓ 后在休息屏加一行'刚才: 8 × 55 kg [调整]', 点开针对刚记录组的 reps/weight 双 stepper 写回 SetRecord (复用 NumStepperField); ② 同屏给 3 秒'已记录 · 撤销' toast 接回现成的 undoLastCompletedSet; ③ 主按钮完成后加 ~300ms 防双击冷却。逻辑两端都在, 只差 UI 触发点。

### 6. [P1] 动作详情页是纯百科: 看不到'我上次推了多重', 也无法把动作加进任何训练——三路独立撞上同一堵墙

**发现路**: ia, library, progress

**证据**: ExerciseDetailSheet (ExerciseLibraryBrowser.swift:615 起) 全文 grep 不到 data.sets——无 last sets、无 PR、无逐动作记录; 库入口打开详情也不传 onAdd (ExerciseLibraryBrowser.swift:547-549), 'Add to workout' 按钮仅 picker 路径渲染 (1021-1040)。个人数据散在三处难达: 播放器 'Last: 55 kg × 8' (开练后才可见)、Progress→Insights 第 6 张卡的无搜索 Menu (InsightsChartsView.swift:571-596, 按组数排序)、History 按天不按动作; 从 History session 详情点动作弹的还是这个图书馆页 (HistoryScreen.swift:1195-1198)。'站在卧推架前查上次多重'是老手最高频查询之一, Strong/Hevy 动作页第一屏就是 History/Records——本 App 有全部数据却没有归属点, 是迁移用户最先对比的地方。ia/library/progress 三路独立发现。

**建议**: ExerciseDetailSheet 加 'Your history' 小节: 最近 3 次 日期·kg×reps + 历史最佳 + 迷你 e1RM sparkline (数据全来自现有 data.sets, 复用 e1rmSeries/weightLabel); 顶栏加'添加到…'菜单走 picker 现有管线。库/训练 picker/session 详情共用此 sheet, 一处改全生效。

### 7. [P1] 今日推荐 (LRU 轮换) 完全不读同屏上方的疲劳模型——'假智能'两天就会被重度用户识破

**发现路**: ia

**证据**: TodayScreen.swift:48-53 suggested 走 pickTodayPlan ('LRU 挑最久没练那张'); fatigueMap (TodayScreen.swift:361-363) 只喂给 MuscleStatusOverviewCard, 推荐路径零引用。today.png 实景: 'Last trained yesterday'、胸部处于 Heavy fatigue, 而下方推荐 Full Body C 第一个动作就是 Bench Press (Decline) 打胸。App 把肌肉状态摆在 Today 最顶上暗示它驱动今日决策, 实际零关联——对 'AI Workout Planner' 定位是信任伤害。

**建议**: 最小改法: pickTodayPlan 给候选按'主打肌群当前 fatigue 值'加惩罚项再轮换; 或至少 WorkoutCard kicker 加推荐理由 ('Glutes & pull 已恢复'), 主肌群 Heavy fatigue 时给 swap 提示。

### 8. [P1] Today 同屏两张人体图用同一亮绿表达相反语义: 上图绿=疲劳别练, 下图绿=今天要练, 下图还没图例

**发现路**: ia, usability

**证据**: today.png: Muscle Status 图例最亮绿=Heavy fatigue (MuscleStatusOverviewCard.swift:93-96 用 accent 绿), 胸/背点成亮绿=刚练完没恢复; 紧挨着的 Today's Workout 卡另一对人体图 (同一 MuscleVisualBlock 组件) 同样亮绿高亮=今日目标肌群, 无图例无标签。第一眼会读成'今天推荐练的正是最疲劳的部位'; 图例文字仅 10pt (MuscleStatusOverviewCard.swift:221)。两路独立发现, 且与'推荐不读疲劳'互相强化误读。

**建议**: 把 fatigue 热图从品牌绿拿出来: Heavy fatigue 用暖色 (橙/红) 渐变, '绿=行动/目标'保留给 WorkoutCard 和 CTA; 只改 MuscleStatusCompute.opacityFor 着色出口和 legendRow 色值。训练卡人体图加一行 'TARGETS/目标肌群' kicker; 图例字号提到 11-12pt。

### 9. [P1] "Train the gaps" 一键把人扔进最多 12 动作、力量重量全为 0 的全屏训练——无预览无确认, 且按钮判定与热图是两套模型

**发现路**: ia, progress

**证据**: TodayScreen.swift:398-459 startGapWorkout 合成 plan 后直接 onStart → fullScreenCover, 与全 App 其他启动路径 (先详情再 Start) 不一致; line 427-435 PlanStep weight: isStrength ? 0 : nil 不回填历史 (对比 RootView.startFreeWorkout:560-572 明确回填)。休假一周回来所有肌群 stale → 一键 12 动作马拉松、每个 0kg 逐个手调。且按钮判定走 muscleLastTrainedMap '3 天没碰' (TodayScreen.swift:377-395), 热图渲染的是 volume+衰减 fatigueMap——today.png 里大片肌肉未恢复按钮却照常亮, 用户无法从图上验证 gaps 是哪几块。

### 10. [P1] 从播放器主界面铅笔编辑并保存, 静默抹掉该动作的逐组重量/次数配置, 默认开的全局同步再把破坏传播到所有 routine

**发现路**: training

**证据**: PlanPlayerScreen.swift:1314-1321 ExerciseInfo 编辑入口构造 EditCurrentStepSheet 时没传 initialSetReps/initialSetWeights (对比 372-379 播放列表右滑入口是传了的) → perSetEnabled=false → Save 时 onSave(..., nil, nil) (2971-2973) → updateCurrentStep 把 setReps/setWeights 置 nil (TrainingSession.swift:802-804)。配了金字塔组 (60/70/80kg) 后从主界面铅笔改个组数, 逐组配置被压平; 1334-1338 的全局同步 (默认 ON) 把压平值写进所有含该动作的计划, 训练结束'保存修改'进一步固化。

### 11. [P1] 计划编辑器改重量对带逐组覆盖的动作不生效——'明明改了 60kg, 开练还是旧重量'

**发现路**: plans

**证据**: 训练中逐组调重会把 setWeights 写回 plan (PlanPlayerScreen.swift:1329, 379-388); 播放时 weightForSet 优先读 setWeights (Plan.swift:30-32)。但 EditStepView 的 Weight 行只写 step.weight (PlansScreen.swift:1596-1607), 既不显示逐组值也不清 setWeights → 编辑器改 55→60, 下次开练仍按旧逐组值走。行摘要 planStepDetailLine (PlansScreen.swift:1347-1356) 也只拼 step.weight, 逐组差异在整个编辑链路不可见。

**建议**: EditStepView 检测 step.hasPerSetOverrides (Plan.swift:41 已有): 有覆盖时展开逐组明细或提示'各组重量不同: 50/55/60 — 点击统一'; 用户改 uniform 值时同步清空 setWeights/setReps。行摘要有覆盖时显示 '50–60 kg' 区间。

### 12. [P1] 训练中不阻止自动锁屏: 力量组无倒计时、几十秒不碰屏手机就锁, 一场训练解锁 12-20 次

**发现路**: training

**证据**: 全仓 grep 无 isIdleTimerDisabled; 力量段是 manual-confirm 不倒计时 (TrainingSession.swift:600), 做组 40-60 秒 + 系统默认 30 秒自动锁屏 → 每组结束屏幕大概率已黑。Live Activity (MasoTrainingLiveActivity.swift) 无 Button/AppIntent, 锁屏上不能点 ✓, 只能解锁进 app。Strong/Hevy 都在训练中保持常亮。

**建议**: playerPresented/有活跃 session 时置 UIApplication.shared.isIdleTimerDisabled = true, session 结束或回 mini-bar 时复位; Settings 训练区加'训练时保持常亮'开关 (默认开)。

### 13. [P1] 中途收工只有'放弃'一条路: End 直接清 session, 没有完成页/PR/分享卡, finishEarly() 是死代码

**发现路**: training

**证据**: End 确认框 (PlanPlayerScreen.swift:479-509) 所有分支都走 store.end() (清空 session, TrainingSession.swift:484-490), 不弹 CompletedView、不触发 trackFinish、workout_finish 事件不发; 为此写的 finishEarly() (TrainingSession.swift:494) 全仓无调用方。健身房关门/没时间跳过最后一个动作是高频场景——做了 80% 的训练拿到的收尾和'误开训练立刻退出'完全相同。

**建议**: session.completedSets 非空时, End 确认框主按钮改'完成并结束'走 finishEarly() (完成页照常展示已完成组/时长/PR), '放弃本次'降级为破坏性次按钮留给零完成组场景。

### 14. [P1] 训练中无法查看当前/下一动作的要领: 缩略图被进度环替换后唯一详情入口消失

**建议**: ExerciseInfo 动作名/BodyHint 加 tap → detailExercise = exercise (sheet 通道已在, PlanPlayerScreen.swift:359-362); 休息屏 RestNextHint 动作名同样接上。零新组件。

### 15. [P1] 自由训练第 2 步没有搜索, 每肌群只露评分 top-6——938 动作库里指名要练的动作大概率选不到

### 16. [P1] 搜索双重失灵: 中文无空格复合词/主流别名/英文连写全部 0 结果; 好不容易命中变体又被折叠在 +N 里且组标题回退英文

**发现路**: library

**证据**: 匹配层: Exercise.swift:373-377 matchesSearch 要求每个查询分词是单 token 子串, zh 名把器械放括号 → 实测'哑铃卧推''杠铃卧推''坐姿划船'均 0 结果 (打空格才有 36 条); 别名全缺: '肩推''腿举/倒蹬''箭步蹲''悬垂举腿'全 0; 英文 'pullup''pushup''ohp''rdl' 全 0。呈现层: 组标题 entryTitle 在组内无纯 base 成员时回退英文 baseName (ExerciseGrouping.swift), 搜索时必然触发——zh UI 搜'上斜'返回一排英文 'Bench Press', 目标藏在 +N 折叠内; 且每敲一个字 expandedGroupKey 被强制收起 (ExerciseLibraryBrowser.swift:487-490)。这些是健身 App 搜索的最高频 query, 每次都失败, 与 prefix(200) 截断叠加后检索通道近乎全废。

**建议**: ① matchesSearch 给 CJK 查询加'贪心切分完全覆盖'分支 (~10 行); ② searchTokens 追加去分隔符连写 token (pullup/chinup 即通); ③ exercises.json 加 aliases 字段人工补 30 个高频口语词; ④ 搜索命中变体的组默认展开、entryTitle 改用 canonical.displayName 保持当前语言、query onChange 收起逻辑对命中组豁免。①②④纯客户端当天可上。

### P2-18. 恢复模型对用户完全黑盒: 无解释入口, 且只数组数不吃强度、所有肌群统一 24h 半衰期——空杆热身 5 组与 100kg 正式组 5 组产生相同疲劳

### P2-20. PR 无完整历史 (硬截 6 条无展开、只显示估算 e1RM 不显真实成绩), 且 isPR 全量 O(n²) 每次进 Progress 都算——数据最多的目标用户最先卡

### P2-21. 容量类指标全部忽略自重动作: 徒手为主的用户看到同一页一半图表说'你几乎没练'、另一半说'你练得很足'

### P2-22. 卡片重排提示永久常驻, 且交互上无法把一张卡移到列表最后

### P2-23. 没有任何体重记录能力: 全 app 唯一体重是 onboarding 的单个静态值, 改一次覆盖一次——增肌/减脂目标的最直接结果指标在 Progress 里缺席

### P2-25. 休息只有'跳过'一个操作: 不能 ±30s、不能暂停, 休息时长训练中无处可改, 归零还响铃自动切下一组

### P2-26. 训练中 '+ Add exercise' 加的力量动作重量恒为 0, 不回填上次记录——不编辑就点 ✓ 会记出 0kg 的 SetRecord 污染统计

## 三、本轮审查盲区(下轮补)

- 付费墙/内购链路零覆盖: Maso 1.5 (build 11) 刚首次开 iapEnabled=true, 但 8 路没有任何一路审查 Pro 锁点选择是否合理、Paywall 文案与转化动线、Restore Purchases、免费额度边界的实际体验——付费开关一打开, 这些就是新的最高风险面 (仅 progress 路顺带提了一句全员 isPro 的 token 成本)。

- 训练中断与恢复零覆盖: app 被系统杀掉/来电/低电量/手机没电时, 活跃 session 和已完成组是否持久化、重开 app 能否续练——健身房场景高频真实事件, 半场训练丢进度是弃用级风险, 没有任何一路验证过。

- Apple Watch / 佩戴场景完全缺席: Strong/Hevy 标配手表端记组和休息计时, 8 路没有任何一路评估 (哪怕作为竞争缺口标注); 与'训练中锁屏/触控目标'两条 P1/P2 同属'器械旁不方便掏手机'的同一用户痛点, 值得作为路线图缺口明确。

- 无障碍只查了触控尺寸: Dynamic Type 放大后训练播放器/图表/onboarding 是否溢出、VoiceOver 对自绘人体图与图表的可读性、Reduce Motion——全部未覆盖; 播放器大量自绘组件是高危区。

- 音频行为未验证: 健身用户几乎必然边听歌边练, 休息结束铃声/动作朗读与用户音乐的混音策略 (ducking vs 打断 Spotify) 无人检查——若每次响铃都掐断音乐, 是每组一次的骂娘点。

- 浅色模式与机型适配未审: 全部截图为同一深色主题同一机型, 浅色模式渲染、小屏 (SE) / iPad 布局无任何证据覆盖。
